# terragrunt_project

Проект для управления виртуальной инфраструктурой в **Proxmox** с помощью **Terraform + Terragrunt + Vault**.

Основная идея проекта — описывать виртуальные машины и инфраструктурные контуры как код, минимизировать дублирование, удобно разделять окружения и сервисы, а также сохранять возможность независимо работать с отдельными VM и кластерами.

---

## Что используется в проекте

- **Terraform** — создаёт и управляет ресурсами в Proxmox.
- **Terragrunt** — надстройка над Terraform для организации структуры проекта, переиспользования конфигурации и разбиения инфраструктуры на независимые state.
- **Vault** — хранение секретов:
  - credentials для Proxmox API
  - cloud-init пользователь / пароль
  - SSH public key
  - другие чувствительные параметры
- **Proxmox** — платформа виртуализации.
- **Cloud-init** — начальная настройка виртуальных машин.

---

## Зачем нужен этот проект

Проект решает несколько задач:

- разделение инфраструктуры по средам (`dev`, `prod`);
- описание отдельных сервисных контуров как набора VM;
- динамическое формирование имён виртуальных машин из структуры каталогов;
- переиспользование общих конфигов для группы VM;
- использование общих секретов в рамках кластера, а не дублирование секретов для каждой VM;
- хранение отдельного Terraform state для каждой VM;
- возможность применять изменения точечно, к одной VM, без необходимости трогать весь контур.

---

## Основные принципы

### 1. Terraform как базовый слой

Terraform используется как основной IaC-инструмент, который создаёт виртуальные машины в Proxmox через переиспользуемый модуль.

### 2. Terragrunt как слой организации

Terragrunt используется для того, чтобы:

- не копировать одинаковую Terraform-конфигурацию по десяткам директорий;
- выносить общую логику в `common.hcl`;
- динамически вычислять параметры из структуры каталогов;
- разбивать инфраструктуру на независимые state;
- удобно масштабировать проект по мере роста количества окружений и VM.

### 3. Отдельный state для каждой VM

Каждая виртуальная машина имеет собственный state-файл.

Это даёт несколько преимуществ:

- можно работать с конкретной VM независимо от остальных;
- изменение одной машины не требует затрагивать весь кластер;
- проще локализовать ошибки и drift;
- проще в будущем вынести state в удалённое хранилище, например в **S3-compatible backend**.

Такой подход удобнее, чем схемы, где несколько VM описываются одним большим Terraform-слоем и обновление одной машины тянет за собой пересчёт или применение ко всему набору.

### 4. Минимизация дублирования

Каждая VM ищет конфиг сначала в своей локальной папке, а затем выше по дереву каталогов.

Это позволяет:

- задавать общие параметры сразу для группы VM;
- переопределять только то, что отличается у конкретной машины;
- не копировать одинаковые `vm-config.yaml` в каждую директорию.

### 5. Использование Vault по кластерному принципу

В проекте используется указатель `vault-path.marker`, который позволяет определить базовую директорию для формирования пути к секретам в Vault.

Это нужно для того, чтобы:

- не создавать отдельный набор секретов для каждой VM;
- использовать общий набор секретов в рамках одного кластера или сервисного контура;
- упростить сопровождение и уменьшить избыточность.

## Пример упрощённой структуры

Ниже приведён простой условный пример:

```text
live/
├── common.hcl
├── dev/
│   ├── vault-path.marker
│   ├── vm-config.yaml
│   └── app/
│       ├── 1/
│       │   └── terragrunt.hcl
│       ├── 2/
│       │   └── terragrunt.hcl
│       └── 3/
│           └── terragrunt.hcl
```

## Как это работает на примере

### Общая логика

В этом примере:

- `common.hcl` содержит общую Terragrunt-логику для всех окружений;
- `dev` и `prod` — разные среды;
- `vault-path.marker` задаёт базовую директорию, относительно которой вычисляется путь к секретам в Vault;
- `vm-config.yaml` задаёт общие параметры сразу для нескольких VM;
- каждая VM имеет собственную директорию и собственный `terragrunt.hcl`.

### Формирование имени VM

Если путь до VM выглядит так:

```text
live/dev/app/2
```

то имя виртуальной машины может быть сформировано динамически как:

```text
app-2
```

Это удобно, потому что:

- имя не нужно вручную прописывать в каждой директории;
- структура каталогов сама становится источником истины;
- проще масштабировать кластеры и добавлять новые узлы.

### Работа с Vault

Если `vault-path.marker` расположен на уровне:

```text
live/dev
```

то секреты могут браться из общего пути вида:

```text
vm/dev
```

Если marker расположен глубже, например на уровне кластера, то секреты можно сгруппировать уже на уровне конкретного сервисного контура.

Это даёт возможность использовать:

- общий SSH key для кластера;
- общий cloud-init user;
- общий cloud-init password;
- и другие параметры, которые не хочется дублировать на каждую VM.

### Работа с конфигами

Если у VM нет локального `vm-config.yaml`, она может использовать конфиг, найденный выше по дереву.

Например:

- общий конфиг для всех VM можно положить в `live/dev/vm-config.yaml`;
- конфиг для конкретного кластера — в `live/dev/app/vm-config.yaml`;
- при необходимости локально в `live/dev/app/2` можно переопределить только отдельные параметры.

## Пример параметров `vm-config.yaml`

```yaml
target_node: "pve"
template_name: "ubuntu-cloudinit-template"
vm_cores: 2
vm_memory: 4096
vm_disk_size: "20G"

disks:
  - slot: scsi1
    size: 10G
  - slot: scsi2
    size: 20G

tags: "dev,app"
vm_net_bridge: "vmbr0"
vm_net_model: "virtio"
vm_disk_storage: "local-lvm"
```

### Что здесь задаётся

- `target_node` — узел Proxmox, где будет создана VM;
- `template_name` — шаблон cloud-init VM;
- `vm_cores` / `vm_memory` — CPU и RAM;
- `vm_disk_size` — размер системного диска;
- `disks` — дополнительные диски;
- `tags` — теги VM в Proxmox;
- `vm_net_bridge` — bridge для сети;
- `vm_net_model` — тип сетевого интерфейса;
- `vm_disk_storage` — storage backend в Proxmox.

## Почему выбран такой подход

Такой подход удобен тем, что:

- инфраструктура разбивается на небольшие независимые элементы;
- можно управлять отдельной VM без необходимости затрагивать весь кластер;
- структура остаётся читаемой даже при росте количества окружений;
- секреты централизованы;
- общие параметры наследуются автоматически;
- state уже логически отделён и готов к переносу в удалённое хранилище.

## Итог

Этот проект показывает подход к организации IaC-инфраструктуры, где:

- Terraform отвечает за создание ресурсов;
- Terragrunt — за структуру, наследование и работу со state;
- Vault — за секреты;
- структура каталогов становится частью логики именования, конфигурации и группировки инфраструктуры.

В результате получается более удобная, масштабируемая и менее избыточная схема управления виртуальными машинами и кластерами.
